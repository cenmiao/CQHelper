## Context

当前项目已实现手动窗口截图功能，核心组件包括：
- `WindowCapturer.cs`: 负责窗口图像捕获
- `ScreenshotSaver.cs`: 负责截图保存为 PNG 文件
- `MainForm.cs`: WinForms 主界面，提供窗口选择和截图按钮

现有功能限制：
- 仅支持手动点击截图按钮
- 无配置持久化机制
- 使用硬编码的 1 秒延时（`Thread.Sleep(1000)`）
- 无定时调度能力

## Goals / Non-Goals

**Goals:**
- 支持用户设置秒级定时间隔（通过输入框）
- 配置持久化到 JSON 文件（`%APPDATA%/CQHelper/config.json`）
- 单个定时任务管理（启动/停止）
- 窗口不存在时自动停止并弹窗提示
- 完全静默截图（无额外用户交互）
- 与现有截图功能无缝集成

**Non-Goals:**
- 不支持多任务并发截图
- 不支持 Cron 表达式或计划时间截图
- 不支持截图后自动发送或上传
- 不修改现有手动截图功能

## Decisions

### 1. 配置文件存储位置

**决策**: 使用 `%APPDATA%/CQHelper/config.json`

**理由**: 
- 符合 Windows 应用程序数据存储规范
- 用户数据与程序目录分离，便于部署和备份
- 避免程序目录写入权限问题

**替代方案**: 
- 程序目录 `./config.json` — 简单但需要写权限
- 用户文档目录 — 可访问性好但路径较长

### 2. 定时器技术选型

**决策**: 使用 `System.Windows.Forms.Timer`

**理由**:
- 直接在 UI 线程触发，无需 `Invoke` 更新界面
- 秒级精度满足需求（~10-100ms 精度足够）
- 与现有 WinForms 架构一致

**替代方案**:
- `System.Threading.Timer` — 高精度但需要处理线程同步
- `Task.Delay + async/await` — 现代但增加复杂度

### 3. 窗口标识持久化

**决策**: 保存窗口标题（`Title`）+ 类名（`ClassName`），启动时重新查找句柄

**理由**:
- 窗口句柄（HWND）在程序重启后失效
- 标题 + 类名组合足够唯一标识大多数窗口
- 实现简单，查找逻辑直接

**替代方案**:
- 仅保存标题 — 可能重复或变化
- 标题 + 进程名 — 更可靠但增加查找复杂度

### 4. 配置保存时机

**决策**: 关键操作后立即保存（选择窗口、修改间隔、启动/停止定时）

**理由**:
- 平衡数据安全和性能
- 避免频繁写磁盘（每次输入都保存）
- 防止程序崩溃时丢失配置

### 5. 窗口验证时机

**决策**: 每次截图前验证窗口是否存在

**理由**:
- 及时发现窗口关闭情况
- 避免无效截图尝试
- 性能影响可接受（截图本身更耗时）

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|---------|
| 窗口标题变化导致无法匹配 | 使用标题 + 类名组合；用户可重新选择窗口 |
| 配置文件写入失败 | 捕获异常并提示；使用 try-catch 包装配置操作 |
| 定时器精度不足 | `System.Windows.Forms.Timer` 对秒级间隔足够；非毫秒级应用可接受 |
| 截图时窗口被遮挡 | 当前实现无法避免；截图内容反映屏幕实际状态 |
| 程序关闭时配置丢失 | 关键操作后立即保存；程序关闭前再次保存 |
| 窗口句柄查找失败 | 弹窗提示用户重新选择窗口 |
